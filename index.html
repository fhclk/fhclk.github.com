微服务间通过消息队列和事件机制异步解耦

发表于 2022-12-12

微服务和微服务间横向集成，即横向东西流量，尽量不要去调同步的web service的服务接口，更适合的方式是通过消息中间件去解耦，把同步请求调用变成走消息中间件，通过消息异步调用，从而到达解耦的目的。
传统在使用消息中间件时，更过的是应用消息中间件做销峰处理、消息发布订阅等。
在微服务中，应用消息队列、EDA事件机制，很重要的点是对各个微服务横向之间进行彻底的解耦。
举例
一个采购的微服务，一个库存的微服务。

  1. 原来的实现方式
    一个订单形成后，通过调用一个实时的API接口，把采购订单同步到库存中心里去。
  2. 微服务架构方式
    采用微服务架构方式的两个变化：
    一是采购订单不需要同步到库存中心，库存中心需要的时候实时查询采购订单。
    二是采购中心首先将采购订单发布到消息中间件和消息队列里面，然后库存中心去订阅消息队列的消息，然后通过异步的方式来获取采购订单，这样才能够彻底的实现采购微服务和库存微服务之间的解耦。

在应用采用微服务架构后，消息机制、事件驱动架构应用的边界往往就是微服务的边界。微服务和微服务之间的横向的调用，要减少实时的接口调用，更多的是利用消息中间件，通过消息机制去做异步的解耦，在微服务应用内部，可以更多的用同步的接口进行同步服务调用，同时在微服务内部，更容易地通过数据库来解决分布式事务的问题。在微服务和微服务之间，仍然会出现分布式事务的问题，通过消息中间件，这类分布式事务问题比较容易采用我们经常说到的Base事务最终一致性的方式来解决。

微服务的拆分不要拆分的太细，拆分的太细，即使使用了消息机制、消息队列，整个异步的集成仍然会相当的复杂。

企业架构底层逻辑-技术架构规划的演进

发表于 2022-11-02

注：“人月聊IT”的学习笔记

传统企业信息化IT建设

也称之为烟囱式的建设模式。在这种建设模式下面，底层统一了IaaS虚拟化资源池，但是上面应用的建设任然是竖井式的一个烟囱一个烟囱的建，每个应用底层有一个小的技术平台，上面分了很多个功能模块，但是这些功能模块是紧耦合在一起的，应用和应用之间往往是通过SOA集成平台，或者是通过ESB总线去实现数据的集成，接口的交互；所以说在最早的企业架构规划的技术架构规划的时候，我们更多的是做的IaaS虚拟化资源池的，它整体的IT物理部署架构的规划，在这个部署架构规划里，我们就会去考虑整个数据库的集群，应用中间件集群，负载均衡，上层核心网交换机等内容，当然对于物理部署架构任然可以细化去考虑它的逻辑架构，物理部署架构更多的只关心你的存储；但是到了IT逻辑规划，往往就会去关心数据库服务器有几台，消息服务器有几台，缓存服务器有几台，上层的APP应用集群有几台，这些就会形成一个类似于功能架构图一样的一个IT逻辑架构规划视图。当然还有一块内容就是狭义的技术架构规划，即在整个应用开发的过程中，对于存储和持久化层，对于应用逻辑层，对于前端展示层，应该用到哪些技术，这些技术体系形成一张完整的分层的技术架构图，这个也可以把它纳入到企业架构规划的技术架构规划里面。比如在数据库持久层，会用到结构化数据库，非结构化数据库，Redis缓存，ActiveMQ消息中间件，应用层用到SpringCloud微服务开发框架，用到API网关，在前端会用到VUE、js；这些东西就会形成一个完整的支撑企业应用开发的一个技术架构体系图，它可以作为技术架构规划的一个部分。

随着SOA、云原生云计算技术的发展，可以看到企业整体的应用建设是逐渐打破烟囱式的建设的模式，逐渐的从传统的单体架构转到微服务架构，对于云计算这一块也是逐渐的从只管IaaS虚拟化资源池，提升到更多的去管共性化的平台层能力，即平台加应用的构建模式。

平台加应用构建模式

平台就是PaaS技术平台，在这个平台里面，传统的单体应用里面，小的技术平台逐渐就没有了，这些共性的技术能力，全部构建到统一的技术平台里面，这个技术平台任然可以按照软件开发生命周期，分为开发平台，执行平台，运维监控平台，比如微服务开发框架、低代码开发可以纳入到开发平台，基于Kubernetes容器云资源调度平台，消息安全日志缓存，各种共性的技术服务能力，可以纳入到执行平台里面；而对于需要去覆盖从资源到服务，从服务到应用的整个大的监控平台，包括实施微服务以后的整个微服务管控治理，统一纳入到整体的治理管控平台里面。

一个底层的共性平台形成后，上层的应用开发就只需去开发一个个业务实现的微服务模块。上层的微服务开发怎样跟底层的平台更好的去集成和协同，怎样去实现微服务，从需求设计开发测试部署交付，整个链条的自动化高度的敏捷和流水线作业，在这个过程中，就会去引入一个覆盖整个微服务端到端开发和交付，实现了持续集成和持续部署能力的一个DevOps平台，当前在去做云原生平台整体规划的时候，也会把DevOps支撑平台纳入到整个平台中的规划体系里面，这样就形成一个完完整整的平台层的架构规划。

随着整个微服务云原生技术的发展和传统IT架构的转型，我们再去给企业做整体的信息化规划，企业架构规划的时候，特别是对于技术架构层面，一定要意识到，它已经不是单纯的去做IaaS基础设施部署架构的规划，而是一定要去做云原生技术中台规划，把云原生技术中台规划做好，才能够真正的实现平台加应用这种快速的构建模式，当然随着整个企业的发展，从IaaS层技术规划转到了PaaS层的规划；如果要去做互联网SaaS对外的应用，还会从PaaS规划转到SaaS平台的规划，大家感觉SaaS层好像是应用层的规划，其实不是，所有的SaaS应用它任然会需要一个SaaS平台，比如多租户怎么设计、多中心怎么设计，这些都是属于SaaS平台层的规划的内容，这些任然应该纳入到整个技术架构规划体系里面，只有这样才能够完整的形成一个完整的技术架构规划的底层逻辑。

企业架构底层逻辑-数据架构的承上启下

发表于 2022-10-31

注：“人月聊IT”的学习笔记

数据架构完全衔接了业务架构和IT架构，它是两者之间的一个关键的衔接点。

对于数据架构里面的数据主题域的分析，数据的业务对象，业务对象的数据和分析，是业务建模阶段要做的事情。

数据架构的两条主线

  • 第一条主线

数据架构里面的数据的逻辑模型、数据的物理模型和数据库设计这些内容自然而然过渡到了IT应用架构规划设计。

数据架构包含数据主题域分析、数据概念模型分析、数据逻辑模型分析和数据物理模型分析。

主题域可以理解成业务价值链的业务域。

概念模型类似于传统IT软件开发里面的业务对象建模。类似于ER的实体关系图。在概念模型阶段，只要识别出核心业务对象和业务对象之间的关系就足够了。

逻辑模型就需要拆分到具体的数据表的粒度。

物理模型需要去对每一个数据库表、每一个字段、字段的类型，字段的约束进行详细的设计。

  • 第二条主线

主数据的规划和建设

通过流程分析、通过CRUD的举证分析，去看每一个数据对象和业务功能之间的耦合关系，耦合关系梳理完后，再结合应用架构，梳理哪些数据对应到哪个系统。通过CRUD分析，做主数据的识别，然后做主数据的分析和主数据的建模，

OLTP过渡到OLAP

传统的BI的架构、数据中台架构。

去看相应的业务系统数据、主数据，如何采集和集成数据，形成统一的ODS库，在ODS库建成后，怎样结合数据的建模形成上层的数据仓库

业务对象和数据对象的区别和联系

  1. 关注者不同

    业务对象的关注者一般是指业务人员，往往对应到相应的业务单据。

    数据对象的关注者是IT人员和技术人员，往往会对应到数据库和数据库表。

  2. 对象本身的转换、映射和颗粒度不同

    业务对象对应到ER图，对应到概念模型。

    数据对象对应逻辑模型和物理模型。

    业务对象只会关心一个粗粒度的大的业务领域的概念。比如采购订单。而数据对象会关注到更细粒度的详细的，对应到数据库表的一些子对象，比如采购订单头、采购订单行、采购订单的分配行等信息。

    业务对象往往是业务术语在谈这个对象，比如在说一个库存管理系统中一个月末的记账统计表。对应到数据对象就变成了库存事务处理表。

    有些数据对象只存在于技术，不存在于业务，如系统配置表等

  3. 抽象和实现上的差异

    从业务对象转换到数据对象时，要更多的考虑IT和技术上面的一个实现，在考虑这个实现的过程中，从技术建模的角度往往就会进行相应的一个抽象。比如，在谈业务对象的时候，在业务人员看框架协议和采购订单是独立的两个业务对象，但在实现的时候，往往对应着同一个数据对象，不管是框架协议还是采购订单，都是存储在采购订单头这个表里面的，到了数据对象这一层，进行了相应的抽象，转成了一个统一的一个对象。

企业架构底层逻辑-从业务战略到业务架构形成

发表于 2022-10-28

注：“人月聊IT”的学习笔记

怎样去形成一个业务架构

  1. 从企业的整个业务战略和业务目标出发。从企业的整个业务战略和业务目标制定后，往往会出现核心的业务价值，即基于企业核心的商业模式和运营模式。分析业务价值时需要使用价值链分析模型。需要考虑去实现核心的业务价值时，一些关键的业务域究竟应该提供什么样的业务能力，业务能力可以理解为对外暴露的API接口服务能力。如果你对相应的垂直行业的业务熟悉，可以通过组织相应的人员开会，进行相应的头脑风暴，得出需要准备什么样的业务能力。
  2. 通过上一步可能得出粗略的业务能力地图，通过业务能力地图可以看到每一个业务应该对外暴露什么样的一些业务能力的API接口，企业的业务价值应该通过各个域什么样的业务能力来支撑。但是每个业务域里面它仍然是一个黑盒。
  3. 细分到每个业务域，每个业务域内部是有很多各业务功能模块，这些业务功能模块本身就会实现相应的能力，这些业务能力不涉及到对外部进行能力的开放和接口。在业务域内，还需要根上游和下游的业务组件打交道，应该分析出功能模块朝上暴露的该有的开放的业务能力，暴露的业务能力可能是业务域种某个功能提供的，也可能是业务域里面多个业务功能组合后提供的。业务域应该具有哪些业务功能，要回到传统的业务流程梳理分析这个维度，需要分析各业务域里面具体有哪些业务流程，对业务流程做二级分析、三级分析，包括到最细节的四级的EPC的流程分析，分析到每一个业务流程的业务活动、业务功能和业务操作步骤等。把业务流程分析完后，每个业务功能点的输入和输出，就可能是这个业务操作形成的单据或数据。当有了业务功能和业务数据后，进行CRUD的矩阵分析，把业务功能列出来，并把业务数据分析出来，分析完成后，要考虑哪些功能应该聚合在一起，达到高内聚低耦合的目的，把拆散到最细粒度的业务功能进行聚合，把这些业务功能聚合完成后，形成业务模块。
  4. 有了业务流程地图后，进行流程分析和业务功能聚合，形成业务架构里面的业务功能图，业务功能图可以从2级、3级、4级展开，形成一个完整的业务架构的业务功能模型图

概念

逻辑思维是指对事物进行观察、比较、分析、综合、抽象、概况、判断、推理的能力。是一种综合理性的思考能力。

如何锻炼逻辑思考能力

  1. 认识到万事万物环环相扣

  2. 凡事都问它的本质是什么

  3. 透过现象看本质，即不要停留在事物的本身，而是要不断追问它是由什么造成的，往前不停地追溯，就总会发现一个再也无法追溯地基础，这个过程叫做逻辑推演

  4. 不断实践，融会贯通

思维和写作/沟通的关系：

写作沟通就是思维的实践过程，一旦缺乏写作沟通的长期实践，人的思维是不可能清晰的。写作的过程也是推动思维的过程，许多想法是在写作的过程中产生的。

基本概念

把传统地单体架构的业务系统，打散为更细粒度的单位，每一个单位都是可以独立的进行需求设计开发测试部署和交付的，每个单位都能够做到独立的自治。这个打散的单位就是微服务，同时拆分出来的每个单位之间，只能通过轻量的rest API接口进行协同或者交互。

核心思想

微服务的核心：大拆小，拆小的每个组件都能够做到独立自治。

一个简单的采购系统。

高可用

微服务是通过对传统的单体应用进行拆分，来实现整个应用它本身的弹性和可扩展能力，但是当对单体应用进行了拆分，对数据库进行了水平和垂直拆分后，一定会引入分布式的问题，基于CAP原理在引入分布式以后，对于整个应用架构底层技术架构，它本身的高可用和高一致性，往往没办法做到同时满足，要么是CP优先，要么是AP优先，两者中只能选一个，因此，在解决一个新问题时，引入新的技术新的架构去解决一个问题的复杂度时，往往会带来其他新的复杂性。

敏捷

错误观念：

应用微服务架构后，整个应用的开发会更加高效更加敏捷。

但是实际中敏捷开发和微服务没有任何关系。

引入微服务架构后，对原来的单体应用做了拆分，做了前后端分离，整个微服务开发团队，前端和后端人和人之间的沟通和协调会跟多，对整个敏捷开发、敏捷管理的要求更高。

复杂度

微服务架构的引入会增加系统的复杂度。

引入微服务后，微服务与微服务之间有集成点，前端和后端之间也有集成点，还和企业原有技术如Paas平台，底层的消息安全缓存等各种技术服务能力之间有集成点。

在有大量接口集成和交付的情况下面，整个微服务应用的开发和交付的复杂度呈现指数级的增长。如果一个团队微服务治理的能力没有跟上，盲目的实施微服务，或者把单体应用拆分成微服务时拆分的太细，往往带来后续的微服务的开发集成、微服务管控、微服务运维一些灾难性的问题。
